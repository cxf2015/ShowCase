# 当前方案 vs 论文方案（简洁版）


## 准确率对比

- 论文方案在准确率上通常更有优势，尤其是在复杂题、难题、以及多个模型能力差异明显的 query 上。
    - 原因是 Decoder-only 模型的 prefill 激活不只是表示“题目语义”，还包含“这题难不难、这个模型大概率能不能做对”的内部信号。
    - 可以把它理解为：模型在正式生成前，脑子里已经形成了一部分“能不能答对”的判断。
- 当前方案使用的是 BERT 类 Encoder 特征，更偏向文本语义、句法和相似度表示，擅长理解“题目在说什么”，但不直接包含“模型能不能做对”的生成能力信号。
    - 因此当前方案更容易学到“题型匹配”，现在模型能学到的是：“这个 query 的语义 + 这个模型”->是否能答对


## bert模型和LLM模型编码的原理对比
✅ Decoder-only（Llama/Qwen）
●预训练任务：自回归生成下一个 token
●天生要学习：推理逻辑、答案合理性、任务难度、自身会不会做错
●Prefill 激活里直接藏着「这道题我能不能答对」的信号（论文叫 correctness direction）
❌ BERT Encoder
●预训练任务：掩码填词（MLM）
●只学习：文本语义、上下文关联、句法结构
●完全不学习 “生成答案” 和 “解题对错”
●它只懂 “题是什么意思”，不懂 “模型能不能做对”

## 泛化性对比

- 对新模型加入模型池的泛化上，当前方案的架构扩展性更好：新增模型时，通常只需要补这个模型的 embedding 和训练样本，旧模型受影响相对较小。
- 当前方案新增模型后的主要风险通常是训练数据分布有轻微漂移，但整体比“所有头一起重训”更可控。
- 论文方案在这点较弱，因为输出头和模型池绑定。新增一个模型，通常需要新增一个头并联合重训，模型越多，多头之间越容易互相干扰。
- 因此论文方案在新模型加入后，更容易出现旧头遗忘、旧模型预测波动和整体收敛变难的问题。
- 需要注意的是，当前方案这里的优势主要是“架构扩展性更好”，并不等于对新模型一定有很强的 zero-shot 效果，最终仍取决于模型 embedding 是否足够表达模型能力。

## 训练对比

- 两边本质上都是“先提一次特征，再训练小头”，所以小头训练时间差不大。
- 真正差别在特征预处理：
- 当前方案：用 BERT/MobileBERT 提特征，速度快，资源要求低。
- 论文方案：用 7B LLM 的 prefill activations 提特征，明显更慢。
- 资源上：
- 当前方案训练特征提取通常 `1-2GB` 显存即可。
- 论文方案 7B prefill 特征提取阶段通常需要模型本体约 `13GB` 加激活开销，实际一般要 `>16GB FP16`。

## 推理对比

- 当前方案不是按 `query+model1`、`query+model2` 这样逐模型重复做完整推理。
- 实际流程是：先对 query 编码一次，再把这个 query 特征和所有候选模型的静态 embedding 拼起来，批量一次性输出所有模型的质量分和成本分。
- 也就是说，逻辑上是在评估 `query-model1` 到 `query-modelN`，但实现上是并行一次算完，不是对每个模型单独跑一遍编码器。
- 当前bert大约可做到 `几十ms` 级别。
- 论文方案 7B prefill 大约是 `100-500ms`。
- 因此论文方案通常比当前方案慢约 `10-50倍`。

## 部署对比

- 当前方案部署简单，适合高并发、低时延场景。
- 论文方案需要额外部署一个可输出 hidden states 的 7B encoder，GPU 依赖明显更强。
- 论文方案 7B 推理显存大致要求：
- `INT4`: `4GB`
- `INT8`: `8GB`
- `FP16`: `16GB+`

## 一句话判断

- 论文方案的核心优势是 Prefill 激活包含 correctness signal，因此在准确率、难题识别和 query 泛化上通常优于基于 BERT 语义特征的方案。
- 当前方案的核心优势是 轻量、稳定、易扩展，更适合低时延和频繁增删模型的工程场景。
- 在模型池扩展性上，当前方案优于论文方案；在 correctness 建模能力上，论文方案优于当前方案。
- 我们可以尝试将我们MobileBERT embedding -> Prefill feature + fisher + mean/lst pooling + PCA(可选)，然后保持当前两个模型头的训练方式（质量分数回归 + output预测）

| **准确率** | **通常更强，尤其在难题和模型能力差异明显的 query 上**<br>• **closes 45.58% oracle gap**<br>• Mean AUC: 0.8560 (Frontier) / 0.8260 (Small) / 0.8817 (Mixed)<br>• 相比最佳语义基线提升 **~5.2%**<br>• P-AUCCC 提升 **14.67%** over model-only Pareto<br>• 优势根因是 prefill 特征含 correctness signal | **暂无论文级实验数据**<br>• 基于 BERT 语义特征，理论上更接近 semantic baseline<br>• 支持 quality + cost 双目标联合预测<br>• 能学到“这个 query + 这个模型是否可能答对”，但主要依赖语义匹配，难题识别能力通常弱于 prefill 方案 |
| **泛化性** | **query 泛化通常更强**<br>• 对复杂任务、跨题型任务更容易迁移<br>• 支持 Frontier / Small / Mixed pool，且可用开源 encoder 预测闭源 API<br>• 但新模型加入时通常要新增输出头并联合重训，模型多了后更易互相干扰 | **模型池扩展性更强**<br>• 新增模型通常只需补模型 embedding 和训练样本，旧模型受影响相对较小<br>• 更适合频繁增删模型的工程环境<br>• 但对新模型的 zero-shot 泛化仍取决于 model embedding 是否足够表达模型能力 |

| **一句话判断** | correctness 建模能力强，通常在准确率上更优 | 工程可用性更强，通常在成本、时延、扩展性上更优 |
